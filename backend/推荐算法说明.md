# 推荐算法说明

本文说明当前后端二推一推荐的实现方式，对应 `backend/service/algo_service.py` 的实现。

这版实现的核心目标是三件事：
- 用两个偏好列表表达“喜欢”和“不喜欢”
- 先算单链条分，再按锚点聚合成组合链
- 返回结果时保留可解释性，但避免把响应体撑得太大

## 1. 接口入口

统一入口：`POST /api/graph/recommend/2to1`

请求体字段：
- `type`: `skill_to_role` / `role_to_company` / `company_to_role`
- `primary_pos_list`: 第一类节点的正向偏好列表
- `primary_neg_list`: 第一类节点的负向偏好列表
- `secondary_pos_list`: 第二类节点的正向偏好列表
- `secondary_neg_list`: 第二类节点的负向偏好列表
- `limit`: 返回的组合链条上限

三个 `type` 只是决定“哪一类节点作为推荐目标”和“组内展开时按什么锚点聚合”，输入格式是统一的。

## 2. 三种推荐类型

- `skill_to_role`: 技能 + 公司 -> 岗位
- `role_to_company`: 技能 + 岗位 -> 公司
- `company_to_role`: 岗位 + 公司 -> 技能

这里的“第一类节点”和“第二类节点”由 `type` 决定。

## 3. 单链条评分

当前单链条的评分规则分成两部分：

单链条始终表示一条 `Company -> Role -> Skill` 路径。不同推荐类型只是把这条路径的“锚点”理解方式换掉。

### 3.1 公司 -> 岗位边 `RECRUITS`

基础分由以下部分组成：
- `urgency`: 普通 = 1，高 = 2，极高 = 3
- `salary`: 取薪资上下限平均值，再除以 10000
- `headcount`: 直接使用数值，作为额外加分项

即：

$$
S_{recruit} = urgency + \frac{salary_{avg}}{10000} + headcount
$$

### 3.2 岗位 -> 技能边 `REQUIRES`

基础分由以下部分组成：
- `weight`: 关系权重
- `proficiency`: 熟练程度系数
-  - 精通 = 1.5
-  - 熟悉 = 1.3
-  - 掌握 = 1.1
  - 了解 = 0.7
- `is_core`: 核心技能额外乘 2
- `skill.category == "软技能"`: 再乘 0.4

即：

$$
S_{requirement} = weight \times proficiency\_multiplier \times core\_factor \times soft\_skill\_factor
$$

### 3.3 节点偏好系数

- 命中正向偏好：乘以 `PREFERENCE_WEIGHT = 3.0`
- 命中负向偏好：乘以 `-NEGATIVE_WEIGHT = -1.5`
- 没有命中偏好：乘以 1

当前实现里，正负偏好都会作为乘法系数直接进入单链条分数，所以偏好命中会显著影响最终排序，负向偏好还会把分数翻成负值。

### 3.4 单链条总分

$$
S_{chain} = S_{recruit} \times factor_{primary} \times S_{requirement} \times factor_{secondary}
$$

代码里同时会保留：
- `base_score = S_{recruit} \times S_{requirement}`
- `preference_score = factor_{primary} \times factor_{secondary}`
- `pair_bonus = 0.0`，当前保留字段，后续如需引入额外配对奖励可以继续扩展

单链条排序按 `score` 从高到低。

## 4. 组合链条评分

当前组合链条会先把单链条按同一个“推荐锚点”聚合，再计算最终分数。

- `skill_to_role`: 按 `role + company` 聚合
- `role_to_company`: 按 `company + skill` 聚合
- `company_to_role`: 按 `role + skill` 聚合

这一步不仅做求和，还会把同一组里的节点和边去重后合并，方便前端直接展示“组内展开”的结果。

组合链条最终分数先采用：

$$
S_{combo} = \frac{\sum S_{chain}}{N}
$$

其中：
- $\sum S_{chain}$ 是该组合下所有单链条分数之和
- $N$ 是链条总数

为了便于后续调参，返回结果里还保留了：
- `total_score`: 单链条分数总和
- `member_count`: 组合内链条数
- `member_scores`: 每条单链条分数
- `single_chains`: 原始单链条预览列表，默认只保留前几条，避免 docs 响应过大

如果按锚点重新拉取后发现同组有更多链条，系统会继续做组内展开，把该组完整成员补进来，再重新计算平均分。

组内展开的目标是：
- 让结果不只显示“最强的一条”，而是显示“一组相关链条”
- 让 `skill_to_role`、`role_to_company`、`company_to_role` 三种模式都能统一处理
- 保留每条成员链的分数和解释，方便后续人工调参

## 5. 你后续最常改的地方

如果你后面要继续调参，通常只需要改这几个位置：
- `PREFERENCE_WEIGHT`
- `NEGATIVE_WEIGHT`
- `PROFICIENCY_MULTIPLIER`
- `URGENCY_SCORE`
- `SALARY_DIVISOR`
- `PAIR_BONUS`（当前保留，但还没有真正参与总分）

## 6. 说明

这份实现目前重点是“可解释、可继续改”。如果你后面要把分数拆成更复杂的多项加权，只需要继续改 `backend/service/algo_service.py` 中的评分函数即可。

另外，这版接口刻意把 `single_chains` 裁成预览数据，避免推荐结果太大导致 docs 或调试页面渲染变慢。
