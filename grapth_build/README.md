# 知识图谱构建与融合流水线

从招聘 CSV 数据构建知识图谱，通过**节点消歧 + 属性增强 + 边收敛**三步完成知识融合。

## 流水线总览

```
data/data.csv → [csvTojson.py] → knowledge_graph.json → [extract.py] → knowledge_graph_final.json
                 (原始图谱构建)    (含噪声的原始图谱)        (知识融合)    (消歧/增强/收敛后的图谱)
```

---

## 第一阶段：原始图谱构建 (csvTojson.py)

将结构化 CSV 数据转化为图结构，定义三类节点和两类关系边。

### 本体设计

| 类型 | Label | 含义 |
|------|-------|------|
| 公司节点 | `Company` | 招聘企业 |
| 岗位节点 | `Role` | 招聘岗位 |
| 技能节点 | `Skill` | 岗位要求的技能 |

| 边类型 | 方向 | 含义 |
|--------|------|------|
| `RECRUITS` | Company → Role | 公司招聘该岗位 |
| `REQUIRES` | Role → Skill | 岗位要求该技能 |

### 节点构建策略

**岗位节点去重**：以「岗位名称」为唯一键，同一岗位名只建一个节点。

**技能节点去重**：以技能名为唯一键（`jobs_map` / `skills_map` 均为字典）。

**公司节点去重**：以「公司名称」为唯一键。

### 多词技能名保护

CSV 中的技能字段以逗号/分号/顿号/空格分隔。部分技能名本身含空格或特殊字符（如 `SQL Server`、`C++`、`Node.js`、`.NET`），直接按分隔符切分会破坏这些名称。为此使用**占位符替换法**：

```
SQL Server → SQL_Server
C++       → CPlusPlus
C#        → CSharp
.NET      → DotNet
TCP/IP    → TCP_IP
PL/SQL    → PL_SQL
```

拆分前替换为占位符，拆分后还原，保护多词技能名的完整性。

### 边权重计算

使用 `Counter` 统计每条边的出现频次：

- **REQUIRES 权重**：同一 (岗位, 技能) 对在数据集中出现的次数（同现次数）
- **RECRUITS 权重**：同一 (公司, 岗位) 对在数据集中出现的次数

权重越高，代表该关系越普遍、越重要。

### 掌握度推断

| 权重 | 掌握度 |
|------|--------|
| < 2 | 了解 |
| 2–4 | 熟悉 |
| ≥ 5 | 精通 |

## 第二阶段：知识融合 (extract.py)

五阶段流水线，对原始图谱执行融合。

### 阶段 1/5：算法模型初始化

**已知大厂列表**：字节跳动、腾讯、阿里巴巴、百度、华为技术有限公司、网易、京东、美团、快手——用于后续规模推断。

**主流技能标准库**（`MAINSTREAM_SKILLS`）：约 80 个核心技能，覆盖编程语言、前端、后端、数据库、大数据、运维、AI/ML 等领域，每个技能有对应的类别（`SKILL_CATEGORY_MAP`）。

**技能别名映射**（`SKILL_ALIAS_MAP`）：约 60 条规则，将大小写/缩写/常见变体映射到标准名，如 `k8s → Kubernetes`、`js → JavaScript`、`golang → Go`。

**语义标签质心**（`TAG_PROFILES`）：7 个预定义标签（高薪、技术大牛多、节奏快、弹性工作、福利好、成长快、核心大厂），每个标签有一段描述其典型特征的关键词文本。使用 `TfidfVectorizer(analyzer='word')` 预先向量化。

### 阶段 2/5：角色消歧

原始数据中同一岗位存在多种变体写法，如「运维工程师」与「运维工程师（5K+/双休/工作轻松）」实质相同。消歧分两轮：

#### 第一轮：规则清洗 + 精确匹配

`clean_role_name()` 清洗岗位名：

1. 去除中文括号内容：`（...）`
2. 去除英文括号内容：`(...)`
3. 去除破折号后的地点/驻场/项目编号后缀：`——XX驻场`、`-DW001` 等
4. 去除常见招聘噪音词：`双休`、`五险一金`、`薪资面议`、`带薪年假`、`可签长期`、`包吃住` 等
5. 清理多余空格

清洗后名称相同的岗位归入同一组。

#### 第二轮：TF-IDF 字符级相似度聚类

对第一轮分组后的规范化名称，使用 **TF-IDF + 余弦相似度** 进行二次合并：

- **向量化器**：`TfidfVectorizer(analyzer='char_wb', ngram_range=(2,4))`
  - `char_wb`：字符级分析，词边界内生成 n-gram，对中文短文本效果好
  - `ngram_range=(2,4)`：提取 2-gram 到 4-gram，捕获中文词组特征
- **合并阈值**：余弦相似度 ≥ 0.85 的两个角色组视为同义，合并到名称较短者（作为规范名）

#### 属性融合

合并后的岗位节点聚合以下属性：

| 属性 | 融合方式 |
|------|----------|
| `name` | 取最短别名作为展示名 |
| `aliases` | 保留所有历史名称，按长度排序 |
| `salary` | 取所有薪资的中位数 |
| `location` | 取所有地点去重后取前 5 个，以 `/` 连接 |
| `education` | 取所有学历要求去重后取前 5 个 |
| `demand_count` | 该岗位在数据集中出现的总次数 |
| `industry` | 由岗位名推断行业 |

### 阶段 3/5：技能消歧

技能名称存在多种变体：大小写差异（`Python` / `python`）、缩写（`k8s` / `Kubernetes`）、拼写变体（`nodejs` / `Node.js`）。

#### 三层标准化策略

**第一层：别名精确映射**

维护 `SKILL_ALIAS_MAP`（约 60 条规则）将常见别名映射到标准名，同时分配类别：

```python
'k8s' → 'Kubernetes'     # 类别: 运维与部署
'js'  → 'JavaScript'     # 类别: 编程语言
'ts'  → 'TypeScript'     # 类别: 编程语言
'golang' → 'Go'          # 类别: 编程语言
```

**第二层：主流技能库大小写不敏感匹配**

将技能名与 `MAINSTREAM_SKILLS`（约 80 个核心技能）逐一比较（忽略大小写），精确命中则标准化并分配类别。

**第三层：编辑距离兜底**

对前两层都未匹配的技能，使用 `difflib.SequenceMatcher` 计算与主流技能库中每个词的相似度：

- `SequenceMatcher.ratio()` 基于最长公共子序列计算相似度（值域 0~1）
- 阈值 ≥ 0.80 时，匹配到最相似的标准名
- 不满足阈值则保留原名，归入「通用技术」类别

#### 技能分类体系

每个技能标准化后分配一个类别：

| 类别 | 示例技能 |
|------|----------|
| 编程语言 | Java, Python, C++, Go |
| 前端技术/框架 | HTML, CSS, Vue, React |
| 后端框架 | Spring, Django, Flask |
| 数据库 | MySQL, Redis, MongoDB |
| 大数据 | Hadoop, Spark, Flink |
| 运维与部署 | Linux, Docker, Kubernetes |
| 云计算 | AWS, 阿里云, OpenStack |
| AI/机器学习 | TensorFlow, PyTorch, NLP, 机器学习 |
| 测试 | 自动化测试 |
| 安全 | 网络安全 |

### 阶段 4/5：公司属性增强

原始图谱中的公司节点仅有名称，通过聚合关联数据推断多维属性。

#### 行业推断 `infer_industry()`

基于公司名称关键词匹配：

| 关键词 | 推断行业 |
|--------|----------|
| 游戏 / game | 游戏 |
| 汽车 / 车 / auto | 智能制造 |
| 银行 / 金融 / 证券 / 保险 | 金融 |
| 医院 / 医疗 / 药 / 健康 | 医疗健康 |
| 教育 / 学 / 培训 | 教育 |
| 移动 / 电信 / 联通 / 政务 / 数字 | 政企服务 |
| 其他 | 互联网软件（默认） |

#### 规模推断 `infer_scale()`

综合考虑公司知名度、招聘体量和标签：

| 条件 | 推断规模 |
|------|----------|
| 属于已知大厂列表 | 大厂 (>10000人) |
| 招聘人数 > 15 或含「国企」标签 | 大型 (500-10000人) |
| 招聘人数 > 5 | 中型 (100-500人) |
| 其他 | 小型 (<100人) |

#### 语义标签推断 `algorithmic_infer_tags()`

使用 **TF-IDF + 余弦相似度** 从公司描述文本推断标签：

1. **构建标签质心**：7 个预定义标签，每个标签有一段关键词文本作为该标签的语义质心
2. **构建公司文档**：公司名 + 地点 + 薪资 + 技能关键词 → 拼接为文档
3. **向量化**：使用 `TfidfVectorizer(analyzer='word')` 向量化公司文档
4. **相似度计算**：计算公司文档向量与 7 个标签质心向量的余弦相似度
5. **Top-3 选择**：取相似度最高的 3 个标签（相似度 > 0.05 才算有效命中）
6. **兜底填充**：不足 3 个时，按优先级填充默认标签：福利好 → 弹性工作 → 成长快

#### 动态描述生成 `generate_description()`

基于公司名、岗位名、地点、技能、薪资，从 3 个自然语言模板中随机选择生成描述文本。

### 阶段 5/5：边收敛

原始图谱中多条边指向语义相同的节点，角色和技能消歧后需要将边重新聚合。

#### REQUIRES 边收敛

- 将旧的 `(role_id, skill_id)` 映射到融合后的新 ID
- 按 `(new_role_id, new_skill_id)` 聚合权重（累加各原始边的权重）
- 根据聚合权重重新计算掌握度：

| 聚合权重 | 掌握度 | is_core |
|----------|--------|---------|
| ≥ 5 | 精通 | true |
| 2–4 | 掌握 | true |
| 1 | 熟悉 | false |

- 权重截断上限为 10，防止极端值

#### RECRUITS 边收敛

- 将旧的 `(company_id, role_id)` 映射到融合后的新 ID
- 按 `(new_company_id, new_role_id)` 聚合招聘人数
- 取该岗位对应薪资的中位数作为边的薪资属性
- 根据招聘人数计算紧急度：

| 招聘人数 | 紧急度 |
|----------|--------|
| > 10 | 极高 |
| 4–10 | 高 |
| ≤ 3 | 普通 |

## 算法汇总

| 阶段 | 算法 | 用途 | 阈值 |
|------|------|------|------|
| 角色消歧 | 规则清洗 + 精确匹配 | 去除噪声后缀，第一轮分组 | — |
| 角色消歧 | TF-IDF (char_wb, 2-4 gram) + 余弦相似度 | 相似角色名聚类合并 | 0.85 |
| 技能消歧 | 别名映射表 | 缩写 → 标准名精确转换 | — |
| 技能消歧 | 大小写不敏感精确匹配 | 主流技能库匹配 | — |
| 技能消歧 | SequenceMatcher (编辑距离) | 拼写变体模糊匹配 | 0.80 |
| 标签推断 | TF-IDF (word) + 余弦相似度 | 公司文档 → 7 个标签质心匹配 | 0.05 |
| 行业推断 | 关键词规则匹配 | 公司名 → 行业分类 | — |
| 规模推断 | 条件规则 | 知名度 + 招聘量 + 标签 → 规模 | — |
| 边收敛 | Counter 聚合 + 中位数 | 融合后边的权重/薪资/紧急度重计算 | — |

### TF-IDF + 余弦相似度原理

**TF-IDF（词频-逆文档频率）**：

- TF（Term Frequency）：词 t 在文档 d 中出现的频率
- IDF（Inverse Document Frequency）：log(文档总数 / 包含词 t 的文档数)，衡量词的区分度
- TF-IDF = TF × IDF —— 高频但普遍的词权重低，高频且稀有的词权重高

**余弦相似度**：

- 将文档表示为 TF-IDF 向量
- cos(θ) = (A · B) / (||A|| × ||B||)，值域 [0, 1]
- 越接近 1 表示两个文档语义越相似

在角色消歧中，使用字符级 n-gram (`char_wb`, 2-4 gram) 而非词级分词，因为中文岗位名通常很短（2-6 字），字符 n-gram 能更好地捕获字形层面的相似性。

在标签推断中，使用词级分析 (`word`)，因为标签质心的关键词文本较长，词级特征能更好地捕获语义。

### SequenceMatcher 原理

基于 **Gestalt 模式匹配**（最长公共子序列）：

- `ratio() = 2 × M / T`，其中 M 为匹配字符数，T 为两字符串总字符数
- 对短文本（如技能名）效果较好，计算速度快
- 适合处理拼写变体（如 `postgresql` vs `PostgreSQL`）

## 运行方式

```bash
# 1. 安装依赖
pip install -r requirements.txt

# 2. 构建原始图谱
cd src && python csvTojson.py

# 3. 执行知识融合
cd src && python extract.py
```

## 输入输出

| 文件 | 阶段 | 说明 |
|------|------|------|
| `data/data.csv` | 输入 | 原始招聘数据，含公司名称、岗位名称、薪资、工作地址、学历要求、专业学术术语 |
| `src/knowledge_graph.json` | 中间产物 | csvTojson.py 输出，含 Company/Role/Skill 节点和 RECRUITS/REQUIRES 边 |
| `src/knowledge_graph_final.json` | 最终输出 | extract.py 输出，节点消歧/属性增强/边收敛完成 |
| `src/knowledge_graph_optimized.json` | 历史产物 | 旧版两阶段流水线输出（仅 Role+Skill，无 Company） |
